01 — Problem Statement

What happens today when a bug ships

Every app has bugs. The real problem is how long customers have to live with them once they're found.

Without OTA, every bug fix must go through the full App Store process:

Current fix timeline — without OTA
🐛
Bug found
Day 0
🔨
Fix written & full build
+2 hours
📋
Submitted to App Store review
+1–3 days
📱
Users manually update
+3–7 days
Total time customers live with the bug: up to 7 days
⭐

1-star reviews stack up

Customers hit the bug repeatedly before a fix reaches them. Each frustrated session is a potential bad review.

🎧

Support tickets flood in

Every user who can't complete an order contacts support. The team spends hours on issues they can't yet fix.

🛒

Lost revenue

Customers who can't order don't wait — they go elsewhere. A checkout bug on a Friday night is expensive.

🚦

Engineering blocked

Engineers see the Crashlytics spike in real time but can't do anything about it until the next release window.

02 — What It Solves

Fix bugs in hours, not days

CodePush OTA lets engineering push a fix directly to every customer's phone — bypassing the App Store entirely.

🏪 Without OTA — App Store only
3–7
days before users are fixed
  • ❌ Submit to Apple & Google for review
  • ❌ Users must manually install the update
  • ❌ ~50% still on the broken version after a week
  • ❌ Reviews and support tickets accumulate
⚡ With OTA — Bitrise CodePush
~2
hours to fix 100% of users
  • ✅ No App Store submission
  • ✅ Completely automatic — no user action
  • ✅ Every active user gets the fix
  • ✅ Customer never knew there was a bug
🔇

Silent delivery

The update downloads in the background while the customer uses the app. It applies on their next launch — no prompts, no interruptions.

🔁

Instant rollback

If a fix introduces a new problem, it can be rolled back to the previous version within minutes — no App Store resubmission.

🏷

Per-brand, per-environment

Each brand (Arby's, BWW, Dunkin', Sonic) has its own update channel. A fix for Sonic won't touch Dunkin'. QA and production are completely separate.

🔒

Secure by design

Deployment keys are stored in Azure Key Vault — not in the codebase. Each update is signed and verified on the device before it loads.

⚠️

One important boundary

OTA only updates JavaScript and UI logic — the part where the vast majority of bugs live. It cannot update native code (camera permissions, hardware modules, OS-level integrations). Those still require a full App Store release. Think of OTA as a fast lane for everything except native changes.

03 — How It Works

Five steps from bug fix to customer's phone

The whole process is automatic once the engineer pushes the fix. Here's what happens in order.

1
Engineer writes and merges the fix
Engineering
The fix is a standard code change. When it merges to the release branch, the CI pipeline kicks off automatically — no manual steps needed.
2
Pipeline builds a small update package
Bitrise CI
Instead of rebuilding the entire app, the pipeline packages only the changed JavaScript — typically 1–2 MB. This takes about 5 minutes, compared to a full native build's 30–60 minutes.
3
Package is uploaded to Bitrise's delivery platform
Bitrise Platform
The update is registered for the target app version — for example, all users on version 1.0.x of the Sonic app. Users on a different version are not affected.
4
App downloads the update silently in the background
Customer's phone
The next time a customer opens the app, it checks for an available update. If one exists, it downloads automatically in the background while the customer uses the app normally — they see nothing.
5
Fix is live on the customer's next app open
Customer's phone
The new code loads silently on their next launch. No prompt. No loading screen. The bug is just gone — and the customer never knew there was one.
~2 hrs
Total time from fix written to all users patched
Compared to 3–7 days via the App Store. The customer experience is completely uninterrupted.
